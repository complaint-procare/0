# FUTURE.md

Локальний backlog майбутніх покращень Complaint CRM.

Цей файл призначений для фіксації ідей і планів. Наявність пункту в списку
не означає, що його вже погоджено до реалізації.

Актуально станом на **26 червня 2026 року**.

## Межі поточного backlog

Безпекові зміни описані тут, тому що вони є головними технічними ризиками,
але їх реалізація потребує окремого прямого погодження користувача:

- перехід із поточної PIN-моделі на server-issued session/Supabase Auth;
- зміна RLS і ролей;
- JWT-захист Edge Functions;
- зміна OAuth state/nonce, CORS і доступу до Google Drive файлів.

Звичайний UI або performance task не є дозволом непомітно змінити ці
контракти.

## Безпека — окремий погоджуваний проєкт

### Server-issued сесія та RLS

Замінити browser-only PIN session на модель, у якій сервер може перевірити
особу й роль кожного запиту. Варіанти:

- Supabase Auth із прив'язаним `public.users.auth_id`;
- окрема Edge Function для PIN exchange, яка видає короткоживучу підписану
  сесію;
- інший централізований identity provider.

Після цього:

- прибрати `anon_all` policies;
- дозволяти операції через `auth.uid()` і server-side role;
- закрити читання `pin_hash`;
- додати route guards як UX-рівень, але не покладатися на них як на security
  boundary;
- додати строк життя сесії, logout/revocation і негайне блокування
  неактивного користувача;
- додати rate limit і блокування перебору PIN.

### Edge Functions та OAuth

- увімкнути JWT verification там, де це можливо;
- не довіряти `uploaded_by`, `actor_user_id` та іншим browser-supplied ID;
- прив'язати Google OAuth flow до admin session і одноразового state/nonce;
- обмежити CORS production origin;
- додати audit для підключення/перепідключення Drive.

### Приватність даних і файлів

- визначити, чи допустимий `anyone with link reader`;
- для приватних вкладень використовувати обмежені permissions або
  server-mediated download;
- визначити правила зберігання персональних даних, retention і видалення;
- не віддавати `complaint_summary_rows`, complaints і user hashes через
  публічні anon policies.

## Резервні копії та відновлення

Підготувати й перевірити:

- регулярний Postgres backup;
- окремий реєстр/експорт Google Drive file IDs і metadata;
- інструкцію відновлення Supabase в новий проєкт;

- моніторинг database size, egress, Edge Function usage і статусу проєкту;
- план дій при вичерпанні квоти або призупиненні Supabase.

## Продуктивність і масштабування

### Серверна пагінація реєстру

Завантажувати з Supabase лише поточну сторінку скарг, а не всю таблицю.
Розмір сторінки може залишитися 50 записів. Окремо отримувати загальну
кількість записів для навігації сторінками.

Очікуваний результат:

- швидше початкове завантаження;
- менший мережевий трафік;
- стабільна робота при зростанні кількості скарг.

### Серверні фільтри та пошук

Виконувати в Supabase фільтрацію за статусом, критичністю, групою, брендом,
типом джерела, мережею, менеджером і датами.

Для пошуку за номером, партією, продуктом, телефоном, описом і рішенням
розглянути Postgres Full Text Search або спеціалізований RPC.

### Точкові detail queries

Не використовувати `list('complaint_change_log')` і
`list('complaint_attachments')` для відкриття однієї скарги. Запитувати
тільки записи з відповідним `complaint_id`, із серверним сортуванням та,
за потреби, пагінацією історії.

### Серверна аналітика

Не завантажувати всі скарги в браузер для підрахунків. Створити SQL view або
RPC, які повертатимуть готові агрегати:

- загальну кількість;
- нові, активні та закриті скарги;
- динаміку за днями, тижнями й місяцями;
- розподіли за статусами, брендами та іншими вимірами.

## Цілісність даних

### Атомарне оновлення та журнал змін

Оновлення скарги й створення записів у `complaint_change_log` виконувати
однією транзакцією через RPC або тригер.

Це має запобігти ситуаціям, коли скарга вже змінена, але історія змін
записалася частково або не записалася зовсім.

Окремо атомарно обробити створення скарги та подію `created`. Для файлів
потрібна saga/compensation модель, тому що Postgres transaction не може
атомарно охопити Google Drive.

### Серверна валідація

Доповнити UI-валідацію правилами на рівні бази даних:

- обов'язкові статус, критичність і група;
- непорожні назва продукту, номер партії та опис;
- коректна комбінація `source_type`, мережі й телефону;
- коректний формат телефону клієнта;
- допустимі значення системних полів.

Для цього можна використовувати `NOT NULL`, `CHECK`, зовнішні ключі та
серверні функції.

### Конкурентне редагування

Додати optimistic locking через `updated_at` або окрему `version`:

- mutation передає версію, яку редагував користувач;
- update проходить лише якщо версія не змінилася;
- при конфлікті UI пропонує оновити дані або порівняти зміни.

## Вкладення

### Узгоджене видалення файлів

Визначити одну зрозумілу модель:

- реально видаляти файл із Google Drive; або
- залишати файл у Drive, але називати дію архівуванням чи приховуванням.

Текст інтерфейсу повинен точно відповідати фактичній поведінці. За потреби
додати можливість відновлення помилково прихованого вкладення.

Hard delete скарги також має мати явну політику щодо папки й файлів у
Google Drive.

### Надійне завантаження

Покращити роботу з кількома файлами:

- прогрес кожного завантаження;
- окремий статус кожного файла;
- повтор невдалого завантаження;
- зрозумілий результат частково успішної операції;
- перевірка розміру й підтримуваного типу до початку завантаження;
- idempotency key або захист від повторного створення metadata при retry;
- cleanup orphan-файлів, якщо upload завершився, а insert metadata — ні.

## Надійність інтерфейсу

### Error Boundary

Query error/retry states уже реалізовані. Залишається додати загальний
React Error Boundary для неочікуваних помилок рендерингу з безпечним
повідомленням, correlation ID і можливістю перезавантажити екран.

### Збереження стану реєстру

Зберігати для користувача:

- пошук і фільтри;
- поточну сторінку;
- сортування;
- вибрані колонки та їх порядок.

Стан можна зберігати в URL або локальних налаштуваннях, щоб він не губився
після відкриття деталей скарги та повернення до реєстру.

### Чернетка нової скарги

Автоматично зберігати незавершену форму локально. Після випадкового
оновлення сторінки або закриття вкладки пропонувати відновити введені дані.

Не зберігати в чернетці вміст файлів без окремо продуманого механізму.

## Аналітика та робочі процеси

### Семантичні типи статусів

Не прив'язувати статистику до редагованих назв на кшталт `Новий`, legacy `Нова` або
`В роботі`. Додати незмінний системний код чи категорію статусу, наприклад:

- `new`;
- `in_progress`;
- `waiting`;
- `closed`;
- `rejected`.

Назву статусу після цього можна буде змінювати без поломки аналітики.

Reopen не повинен вибирати «перший активний незакритий» статус без явного
системного коду або налаштування цільового статусу.

### Узгоджена семантика карток

Зараз value картки показує всю відфільтровану вибірку, а delta порівнює
останні 7 днів із попередніми 7 днями. Потрібно:

- або показувати value за тим самим періодом;
- або явно підписати, до якого періоду належить delta;
- переглянути tone для «Закриті», де збільшення кількості закритих скарг
  не обов'язково є негативним.

### Розширена аналітика

Можливі доповнення:

- фільтр і розподіл за групами скарг;
- середній та медіанний час закриття;
- кількість перевідкриттів;
- проблемні партії, товари, бренди та мережі;
- порівняння з попереднім періодом;
- експорт вибірки й агрегатів у CSV або Excel.

### SLA та дедлайни

Додати строки першої реакції та вирішення залежно від критичності.

У реєстрі й деталях показувати:

- кінцевий строк;
- час, що залишився;
- наближення до порушення;
- прострочення.

Аналітика може показувати частку скарг, опрацьованих у межах SLA.

## Автоматизовані перевірки

### Тести

Додати кілька рівнів тестування:

- unit-тести форматування, телефонів, нормалізації та аналітичних функцій;
- компонентні тести форм і валідації;
- інтеграційні тести міграцій, RPC, тригерів і журналу змін;
- E2E-сценарії створення, редагування, закриття та перевідкриття скарги.

### CI

Перед frontend deploy автоматично виконувати:

1. TypeScript-перевірку;
2. unit та інтеграційні тести;
3. production build;
4. перевірку створених SPA route pages.

Для Supabase-змін додати перевірку міграцій і базових RPC на тимчасовій або
локальній тестовій базі.

Окремо перевіряти security invariants:

- anon не читає protected tables;
- user не виконує admin mutation;
- forged actor/user ID не дає доступ;
- direct update не обходить audit;
- закрита скарга не змінюється забороненим шляхом.

## Доступність і UX

### Діалоги та складні контролери

- `role="dialog"` і `aria-modal`;
- focus trap;
- повернення фокуса на trigger після закриття;
- коректна клавіатурна навігація autocomplete та multi-select;
- явні labels/combobox/listbox semantics;
- перевірка контрасту кастомних HEX-кольорів.

### Небезпечні bulk/import операції

Поточний каталог продуктів уже має Excel preview, шаблон, експорт усіх товарів і матчинг імпорту за `id`/SKU/name+brand.
Наступні покращення лишаються відкритими:

- транзакційне пакетне оновлення або RPC для імпорту, щоб уникати частково застосованих змін;
- окреме явне підтвердження для опції «видаляти товари, яких немає у файлі»;
- детальний звіт про часткові помилки із рядками Excel;
- автоматична пропозиція експорту стану перед destructive import;
- відновлення або rollback-сценарій для помилкового масового імпорту.

## Завершено

### 26 червня 2026 — нормалізація статусів і критичності

- активні статуси скарг нормалізовано до канонічного набору з 9 назв;
- legacy-статуси `Нова`, `Закрита`, `Відхилена` remap-ляться на `Новий`, `Закрито`, `Відхилено`, а дублікати вимикаються;
- рівень критичності `Критична` об'єднано з `Висока` з remap скарг і `complaint_change_log`;
- seed оновлено: нові бази отримують канонічні статуси й 4 активні рівні критичності без `Критична`;
- кольори наявних статусів і `Висока` зберігаються міграціями.

### 24 червня 2026 — швидші дії в реєстрі та UX-оновлення

- статуси в реєстрі стали клікабельними: зміна статусу відкриває confirmation dialog без окремої колонки «Змінити статус»;
- у деталях скарги рішення/відповідь можна редагувати через іконку олівця;
- кнопка **«Відкрити»** і статуси отримали liquid glass-ефект, після ітерації текст зроблено не жирним;
- у реєстрі додано лічильник унікальних переглядів з іконкою ока.

### 24 червня 2026 — перегляди скарг і Supabase Free keepalive

- додано таблицю `complaint_views` з унікальністю `complaint_id` + `user_id`;
- додано RPC `record_complaint_view` і `get_complaint_view_counts`;
- відкриття деталей скарги рахується як один перегляд на користувача;
- додано `.github/workflows/supabase-keepalive.yml`, який щодня опівночі Europe/Kyiv робить легкий REST-запит, щоб Supabase Free-проєкт не засинав.

### 24 червня 2026 — каталог продуктів, пошук та Excel

- у продуктах додано пошук за назвою, товарним ID, SKU/штрихкодом і брендом;
- додано необов’язковий `products.external_id`, який у UI показується як **ID**;
- Excel-шаблон продуктів отримав колонку `id`;
- імпорт Excel оновлює товари за збігом ID, SKU або name+brand;
- додано експорт усіх товарів в Excel;
- форма редагування товару дозволяє змінювати ID, назву, SKU/штрихкод і бренд.

- створено спільний `ComplaintFormState`;

- додано зрозуміле повідомлення, технічні деталі та retry;

  помилці Supabase;
- помилка фонового оновлення не приховує останні успішні дані;
- pattern підключено до реєстру, створення, деталей, аналітики,
