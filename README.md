# Реєстр скарг (Complaint CRM)

CRM-подібний веб-додаток для централізованого реєстру скарг клієнтів з конструктором сутностей та полів. Збудовано на Vite + React + TypeScript + Tailwind. Локальний режим зберігає всі дані в IndexedDB браузера (без серверних залежностей).

Інтерфейс — у світлій BEXET-стилістиці: біла бічна панель, чорний активний стан, м'які пастельні бейджі статусів і критичності, заокруглені картки.

## Швидкий старт

```powershell
npm install
npm run dev
```

Відкрийте `http://localhost:5173`.

## Навігація

Головна бічна панель має три пункти:

- **Скарги** — реєстр скарг
- **Аналітика** — метрики, графіки, фільтри
- **Налаштування** — розкривається у вкладеному дереві з підпунктами: Клієнти, Бренди, Продукти, Торгові мережі, Користувачі, Сутності, Поля, Загальні

Старі URL (`/clients`, `/brands`, `/admin/users` тощо) автоматично редіректять на нові `/settings/*`.

## Що реалізовано (MVP)

- **PIN-вхід** з рольовою ідентифікацією та локальною сесією.
- **Реєстр скарг**: таблиця для desktop, картки 1×N для mobile, фільтри, пошук.
- **Створення/редагування скарг** із обов'язковими полями ТЗ.
- **Деталі скарги**: повний перегляд, редагування, preview фото/відео, завантаження, видалення.
- **Історія змін**: фіксуються зміни полів, статусу, додавання й видалення файлів, перевідкриття.
- **Статуси**: Нова, В роботі, Очікує відповідь клієнта, Очікує ВКЯ, Закрита, Відхилена. Закриті скарги не редагуються — лише перевідкриваються.
- **Критичність**: 5 рівнів, «Висока» та «Критична» візуально виділені.
- **Вкладення**: будь-які файли, локально через IndexedDB (як стенд-ін Google Drive). Підтвердження перед видаленням.
- **Аналітика** (`/analytics`):
  - 4 картки метрик (Всього / Нові / В роботі / Закриті) з трендом ±% за 7 днів проти попередніх 7
  - барчарт динаміки з перемикачем періоду День (14 днів) / Тиждень (8 тижнів) / Місяць (12 місяців)
  - розбивка за статусами і брендами зі шкалами %
  - **multi-select фільтри** з пошуком: Бренди, Товари, Статуси, Критичність, Торгові мережі, Менеджери — у кожному можна обирати кілька значень одночасно
- **Налаштування** (`/settings`): єдиний хаб для довідників і адмін-інструментів.
- **Конструктор сутностей** (Налаштування → Сутності): створення кастомних довідників, керування активністю та видимістю.
- **Конструктор полів** (Налаштування → Поля): додавання полів будь-якій сутності, типи — text/textarea/number/date/boolean/select/reference. Системні поля не можна видалити, лише приховати.
- **Скидання локальних даних** (Налаштування → Загальні).

## Архітектура

```
src/
├── components/
│   ├── Layout.tsx          # головна оболонка з sidebar (групи + підпункти) та хедером
│   ├── SettingsLayout.tsx  # обгортка для /settings/* з breadcrumb-заголовком
│   ├── Badges.tsx          # пастельні pill-бейджі статусів/критичності
│   ├── ui/
│   │   ├── primitives.tsx  # Button, Card, Input, Select, Field, Toggle…
│   │   ├── multi-select.tsx # multi-select з пошуком, чіпами і лічильником +N
│   │   ├── dialog.tsx, toast.tsx, autocomplete.tsx, source-picker.tsx
│   └── admin/              # CRUD-обгортки для довідників
├── lib/
│   ├── db.ts        # шар даних (IndexedDB через localforage)
│   ├── auth.tsx     # PIN-вхід, сесія, ролі
│   ├── complaints.ts # бізнес-логіка скарг + історія
│   ├── seed.ts      # початковий датасет
│   ├── types.ts     # моделі
│   └── utils.ts     # хеш PIN, формат дат
└── pages/
    ├── Login.tsx, Complaints.tsx, NewComplaint.tsx, ComplaintDetails.tsx
    ├── Analytics.tsx       # сторінка аналітики з фільтрами і графіками
    └── admin/              # Clients, Brands, Products, Networks, Users, Entities, Fields, Settings
```

Шар даних (`lib/db.ts`) ізольовано від UI — для переходу на Supabase достатньо переписати реалізацію функцій `list/getById/insert/update/remove` без зміни сторінок.

## Дизайн-токени

Палітра у [tailwind.config.js](tailwind.config.js):

- фон: `hsl(40 12% 95%)` (теплий світло-сірий)
- бічна панель і картки: `#FFFFFF`
- `primary` / активний стан: `hsl(0 0% 8%)` (майже-чорний)
- pill-кольори в [src/index.css](src/index.css): `.pill-good` (м'ятний), `.pill-warn` (пастельний бурштин), `.pill-bad` (м'який рожевий), `.pill-neutral` (нейтральний сірий)
- кнопки — `rounded-full` капсули; картки — `rounded-2xl` з легкою тінню `shadow-card`

## Запуск через GitHub за прямим посиланням

Проєкт сконфігуровано з `base: './'` у `vite.config.ts`, тому збірка працює як з GitHub Pages, так і з будь-якого статичного хостингу.

```powershell
npm run build       # → dist/
npm run preview     # локальний попередній перегляд
```

Щоб опублікувати на GitHub:

1. Створіть репозиторій на GitHub та запуште код.
2. Додайте GitHub Actions workflow для деплою `dist/` на Pages (див. AGENT.md).
3. У `vite.config.ts` за потреби вкажіть `base: '/<repo>/'`.

## Запуск через Telegram-бот

Telegram WebApp може відкривати ваш сайт як міні-додаток. Після публікації на GitHub Pages (або іншому HTTPS-хостингу):

1. Створіть бота через [@BotFather](https://t.me/BotFather) → `/newbot`.
2. Налаштуйте Menu Button: `/setmenubutton` → оберіть бота → введіть URL вашого додатку.
3. Або підключіть Mini App: `/newapp`, додайте URL `https://<owner>.github.io/<repo>/`.

Додаток відкриється всередині Telegram як веб-додаток. У головному файлі додатку можна підключити Telegram WebApp SDK для отримання `initData` і автоматичної ідентифікації користувача (не вмикайте, доки не реалізовано серверну валідацію).

## Перехід на Supabase (наступний крок)

1. Створіть Supabase-проєкт і таблиці згідно зі схемою з [AGENT.md](AGENT.md#модель-даних-supabase).
2. Замініть `src/lib/db.ts` на обгортки `supabase-js`.
3. Винесіть бізнес-логіку (`createComplaint`, файли) в Edge Functions.
4. Підключіть Google Drive через service account (тільки в Edge Functions).

Frontend код, маршрути та компоненти змін не потребують.
